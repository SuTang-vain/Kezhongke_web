## 1. 产品定位

壳中客 Kezhongke 官方网站是围绕 AI、多智能体协作、人文技术实践与社区共建展开的官方门户。

网站不是传统品牌展示页，也不是单纯的技术博客，而是一个可感知的生态入口：用户进入后，应立即感受到品牌的思想气质、知识体系、人才路径与共建机制。

核心关键词：

- 共鸣：人与知识、人与系统、人与社区之间产生连接。
- 协作：多智能体、开发者、创作者、产业实践者共同工作。
- 共建：开放原型、开放议题、开放贡献路径。

## 2. 产品目标

### 2.1 品牌目标

- 建立"壳中客 Kezhongke"的明确品牌识别。
- 通过克制、高对比、温暖的人文视觉语言，区别于常见 AI 工具站和普通 SaaS 官网。
- 让用户在首屏理解：壳中客关注 AI、协作、技术文明与开放共建。

### 2.2 内容目标

- 将教程培养、人才培养、产品共建三大体系转化为清晰的网站入口。
- 支持长期沉淀文章、案例、项目、贡献者档案和原型资产。
- 让网站具备内容扩展能力，而不是一次性静态页面。

### 2.3 转化目标

- 引导用户阅读深度内容。
- 引导开发者、设计师、研究者进入共建项目。
- 引导产业实践者查看案例并建立联系。
- 为后续社区、CMS、项目看板和贡献体系预留扩展空间。

## 3. 用户画像

### 3.1 学术探索者

关注 AI、社会学、认知科学、人机关系等议题。希望获得系统化知识入口，而不是碎片化文章列表。

核心需求：

- 理解壳中客的思想框架。
- 浏览知识体系与研究方向。
- 阅读深度文章、研究札记和人文观察。

### 3.2 产业实践者

关注 AI Agent、自动化工作流、组织协作效率与真实产业落地。

核心需求：

- 看到真实案例。
- 理解技术方案与资源整合方式。
- 判断壳中客是否具备合作价值。

### 3.3 社区共建者

包括开发者、设计师、研究者、创作者和学生群体。希望在开放环境下参与产品原型、组件实验、内容共创或项目协作。

核心需求：

- 看到正在开放的项目和议题。
- 理解自己可以如何参与。
- 查看贡献者路径和社区机制。

## 4. 设计原则

### 4.1 情绪基调

整体应呈现：

- 克制
- 温暖
- 人文
- 精确
- 有思想深度
- 不像普通 SaaS 官网
- 不像炫技型 AI 工具站
- 不用强烈赛博朋克或过度科技感视觉

### 4.2 视觉原则

- 高对比文字，保证阅读清晰。
- 动效像"呼吸"和"共鸣"，不做高频闪烁、扫描线和噪声堆叠。
- 图形系统应表达"连接、生长、路径、共建"。
- 首屏必须直接建立"壳中客 Kezhongke"品牌识别，而不是只出现抽象口号。
- 页面不依赖大段说明文字解释功能，应通过结构、标题、入口与状态让用户自然理解。

## 6. 页面需求

## 6.1 首页 Home

### 目标

让用户在 10 秒内理解：

- 壳中客是什么。
- 它关注什么问题。
- 用户可以从哪里进入生态。

### 页面模块

1. 首屏共鸣叙事 Hero
2. 三大体系入口
3. 最新刊物 / 项目动态
4. 代表性案例或共建原型预览
5. 加入社区 CTA

### 首屏内容

必须包含：

- 品牌名：壳中客 Kezhongke
- 主标语：建议控制在 16 个中文字符以内
- 品牌说明：60-100 字
- 主 CTA：进入生长体系
- 次 CTA：浏览共建工坊

示例文案方向：

> 在技术与文明之间，重建人的共鸣。

说明文案方向：

> 壳中客关注 AI、多智能体协作与人文技术实践，连接知识、人才与产品共建，让技术系统回到可理解、可参与、可持续的公共语境中。

### 视觉要求

- 背景可使用粒子、文字场、连接线、低密度图谱或生成式视觉。
- 不要使用普通渐变大背景作为唯一视觉。
- 首屏高度建议 `min-height: 88vh`，底部露出下一屏内容，提示页面可继续浏览。
- 首屏品牌名必须是第一视觉中心之一。

### 交互要求

- 鼠标移动时，背景产生轻微扩散、连线响应或文字边缘扰动。
- 移动端改为自动缓慢动态，不依赖鼠标。
- 支持 `prefers-reduced-motion`，关闭大面积动效。
- CTA hover 只做轻微位移、描边、色彩或阴影变化。

### 验收标准

- 桌面首屏无需滚动即可看到品牌名、说明和 2 个 CTA。
- 移动端首屏文字不溢出、不遮挡、不小于 15px。
- 首屏视觉氛围符合"温暖、人文、技术"的气质。
- 禁用动效后品牌文案与导航仍清晰可读。

## 6.2 三大体系入口

### 模块

| 体系   | 页面      | 说明                |
| ---- | ------- | ----------------- |
| 教程培养 | Grow    | 展示优质的教程内容、教程共建    |
| 人才培养 | Path    | 展示成长路径、试点项目和贡献者档案 |
| 产品共建 | Atelier | 展示开放原型和共建议题  |

### 每个入口应包含

- 名称
- 一句话解释
- 代表性视觉符号
- 进入按钮
- 当前动态，例如"12 个主题""6 个试点项目""24 个开放原型"

### 设计要求

- 数据从统一配置读取，不硬编码在组件内部。
- 卡片可点击，但按钮也必须可独立聚焦。
- Hover 时仅做轻微位移、描边、背景变化，不做剧烈缩放。
- 移动端一列展示，桌面端三列或非对称布局。

## 6.3 生长 Grow｜教程体系

### 目标

展示系统化的教程内容，让用户按主题和难度找到学习入口。

### 页面模块

1. 页面标题与简短说明
2. 主题分类筛选器
3. 教程卡片列表
4. 学习路径推荐
5. 相关内容推荐

### 教程卡片应包含

- 标题
- 分类：AI / Agent / 社会学 / 设计 / 工程
- 难度：入门 / 中级 / 进阶
- 简短摘要
- 关联教程链接

### 设计要求

- 教程卡片采用统一布局，分类用色彩或图标区分。
- 筛选器支持按分类和难度组合筛选。
- 学习路径以线性流程图或步骤条展示，不做复杂图谱。
- 移动端列表一列展示，筛选器可横向滚动。

## 6.4 路径 Path｜人才与案例体系

### 目标

展示"从学习者到贡献者，再到产业实践者"的成长路径。

### 页面模块

1. 路径总览
2. 试点项目墙
3. 贡献者档案
4. 影响力路径图
5. 加入路径 CTA

### 案例卡片应包含

- 标题
- 领域
- 挑战与问题
- 方案与方法
- 结果
- 使用技术
- 封面图
- 参与贡献者

### 贡献者卡片应包含

- 姓名
- 角色
- 简介
- 头像（可选）
- 技能标签
- 贡献内容
- 外链（GitHub / 个人站 / X）

### 设计要求

- 不做排行榜式竞争视觉。
- 影响力表达应偏向"贡献轨迹"，而不是单纯积分。
- 案例页结构化展示：背景、挑战、方法、结果、参与者。
- 贡献者头像和档案风格应克制，不做社交媒体式强曝光。
- 支持缺少头像、缺少外链等不完整数据状态。

### 验收标准

- 用户能在 30 秒内理解人才体系的进入方式。
- 案例卡片能清楚表达领域、问题、方案和结果。

### 页面模块

### 原型卡片应包含

### 共建议题应包含

- 标签
- 负责人（可选）
- 相关链接（讨论 / 仓库 / 文档，可选）

### 设计要求

- 原型卡片支持按状态和标签筛选。
- 外链必须有明确图标与视觉提示，让用户知道会离开当前站点。
- 共建议题用状态色彩区分不同阶段。
- 空状态应引导用户查看其他标签或提交新议题。

## 6.6 刊物 Journal

### 目标

承载深度内容，形成长期品牌资产。

### 内容类型

- 项目复盘

### 文章列表卡片应包含

- 标题
- 描述
- 发布日期 / 更新日期
- 作者
- 分类
- 标签
- 封面图（可选）
- 预计阅读时间

### 文章页设计要求

- 阅读宽度控制在 `680px-760px`。
- 正文字号桌面端 17-18px，移动端 16px。

- 支持目录、脚注、代码块、引用、图片说明。
- 支持上一篇、下一篇、相关专题。
- 代码块可横向滚动，但页面整体不能横向滚动。

## 7. 设计系统

### 7.1 色彩

主色：

| 名称 | 色值 | 用途 |
| --- | --- | --- |
| 黑 | `#1A1A1A` | 主文字、深色背景 |
| 暖白 | `#F8F6F1` | 页面背景 |
| 纸白 | `#FFFEFA` | 内容区域背景 |

强调色：

| 名称 | 色值 | 用途 |
| --- | --- | --- |
| 琥珀橙 | `#D94B2B` | CTA、当前状态、关键节点 |
| 深赤 | `#8F2E1F` | 强调 hover、深色模式点缀 |

辅助色：

| 名称 | 色值 | 用途 |
| --- | --- | --- |
| 石墨灰 | `#3A3A3A` | 次级标题 |
| 中灰 | `#7A7A7A` | 辅助文字 |
| 线框灰 | `#D8D2C8` | 分割线、边框 |

使用规则：

- 强调色只用于 CTA、当前状态、关键节点。
- 大面积背景优先使用暖白或黑。
- 避免全站变成单一橙色调。
- 文字和背景对比度需满足 WCAG AA。

### 7.2 字体

中文标题：

- Source Han Serif / 思源宋体
- fallback：Noto Serif SC, Songti SC, serif

中文正文：

- Source Han Sans / 思源黑体
- fallback：Noto Sans SC, PingFang SC, sans-serif

英文与代码：

- Inter
- JetBrains Mono / SFMono-Regular

### 7.3 排版

建议字号：

| 层级 | 桌面 | 移动 |
| --- | --- | --- |
| H1 | 48-72px | 34-42px |
| H2 | 32-44px | 26-32px |
| H3 | 24-30px | 21-24px |
| 正文 | 17-18px | 16px |
| 辅助文字 | 14-15px | 13-14px |

规则：

- 不使用 viewport 单位直接控制字体大小。
- 字间距默认 0，避免负字距。
- 长文正文行高建议 1.75-1.9。
- 紧凑组件内标题不要使用过大字号。

### 7.4 圆角与容器

- 普通卡片：8px。
- 按钮：6px 或 8px。
- 输入框：6px 或 8px。
- 大面积页面区块不做浮动卡片。
- 不使用大量胶囊形按钮，除非是标签或筛选器。
- 不嵌套卡片，避免页面变成多层容器堆叠。

### 7.5 图标

- 建议使用线性图标风格（如 Lucide）。
- 工具、外链、筛选、搜索、关闭、展开等按钮优先使用图标。
- 不熟悉的图标需要 tooltip 或文字标签。
- 外链图标需明确表达会离开当前站点。

## 9. 响应式要求

断点建议：

| 断点 | 范围 |
| --- | --- |
| mobile | `< 768px` |
| tablet | `768px - 1023px` |
| desktop | `>= 1024px` |
| wide | `>= 1440px` |

移动端重点：

- 首页文案不能堆成大段。
- 导航点击区域不小于 44px。
- 长文阅读不出现横向滚动。
- 卡片一列展示。
- 筛选器可横向滚动，但必须有边界阴影或视觉提示。

## 10. 动效规范

动效原则：

- 慢启动、轻反馈、可中断。
- 不使用高频闪烁。
- 不让动效成为信息获取障碍。
- 背景动效负责氛围，前景动效负责状态反馈。

参数建议：

| 类型 | 时长 |
| --- | --- |
| 普通 hover | 150-220ms |
| 页面过渡 | 300-500ms |
| 抽屉打开 | 240-360ms |
| 大背景呼吸动效 | 6-12s 循环 |

推荐 easing：

```css
cubic-bezier(0.22, 1, 0.36, 1)
```

必须支持：

```css
@media (prefers-reduced-motion: reduce) {
  * {
    animation-duration: 0.001ms !important;
    transition-duration: 0.001ms !important;
  }
}
```

## 11. 可访问性要求

- 所有按钮、链接、筛选器可键盘访问。

- 当前导航项必须有可感知状态。
- 抽屉、菜单、弹窗需要正确处理焦点锁定与 Esc 关闭。
- 图标按钮必须提供可访问标签。

## 14. 风险与约束

### 14.1 视觉风险

风险：过度抽象导致用户无法理解网站实际功能。

处理方式：

- 首屏保留明确品牌说明和 CTA。
- 三大体系入口使用清晰标题和一句话解释。
- 每个页面都保留具体内容预览。

### 14.2 内容维护风险

风险：内容写死在组件内，后期难维护。

处理方式：

- 所有文章、案例、原型、贡献者使用结构化数据。
- 展示组件与数据源解耦。
- 从 MVP 开始预留 CMS 迁移空间。
